# 제안: verify → code 자동 수정 루프

> 상태: 규칙화 완료 → code-rules §10 · code SKILL fix 모드 · ddalkak 수렴 루프로 반영됨
> 대상 단계: ③code(내 담당) · 목적: "생성 → 검증"에서 멈춘 파이프라인을 "생성 → 검증 → 자동 수정 → 재검증(수렴)"으로 닫는다.

> **갱신(2026-07-08)**: §5·§6에서 "혼자 못 가는" 사유로 꼽았던 매칭 계약 불일치가 해소됐다.
> visual-verify v2에서 `match.mjs`가 앵커를 **인덱스 경로**(`children[i].children[j]`)로 통일했고,
> 이는 code-rules §4-3과 정확히 일치한다. 또 `visual.json`이 `checks.items[]`(속성 단위 기대/실제)와
> `matches[]`(`nodePath`↔`dkPath`↔`dataDk`)를 함께 담으므로, 두 배열을 `nodePath`로 조인하면
> verify를 건드리지 않고도 소스를 `data-dk`로 특정할 수 있다. 따라서 4-2의 위치 특정이 성립하고
> fix 모드를 code-rules §10으로 규칙화했다. 아래 4~8장은 그 설계 근거로 남긴다.

## 1. 배경 & 문제

지금 파이프라인은 verify가 **불일치를 찾기만** 한다. 고치는 건 사람이 수동으로 하거나 code 전체를 재생성하는 수밖에 없다.
- 실측(전사이클 리포트): 진성 불일치 4건이 `파일:줄 + 기대→실제`까지 정확히 짚였지만, **그걸 코드에 반영하는 자동 경로가 없다.**
- 결과: verify는 "틀렸다고 알려주는 린터"에서 멈춰 있고, 파이프라인 가치가 반감된다.

## 2. 목표 / 비목표

- **목표**: verify 산출물을 입력으로, **짚어진 지점만** 고치고 다시 검증해 통과까지 수렴시키는 code의 "fix 모드".
- **비목표**: verify 로직 변경(순수 심판 유지), 전체 재생성, 사람 게이트(plan) 대체.

## 3. 개요

```
code 생성 → verify → 통과? ─ 예 → 끝
                       └ 아니오 → [fix 모드] 짚어진 요소만 패치 → verify → …(수렴/중단까지)
```

## 4. 설계

### 4-1. 입력 — verify 산출물
`.ddalkak/reports/<name>.<breakpoint>.visual.json`의 `checks.items[]`를 소비한다.
각 항목: `{ nodePath, kind, expected, actual, status: "fail", severity }`
→ "어느 노드의 어느 속성이 기대 X인데 실제 Y"를 속성 단위로 준다(예: `kind=radius, expected 12, actual 9999`).

### 4-2. 위치 특정 — nodePath → 소스 요소 (핵심 난관)
verify가 준 `nodePath`를 소스 코드의 어느 요소인지 확정해야 한다.
- **data-dk 있으면**: `data-dk="<nodePath>"`로 정확히 특정(신뢰도 high). ← 이 루프가 data-dk를 필요로 하는 이유.
- **없으면**: 컴포넌트/텍스트로 추정(신뢰도 low, 오수정 위험) → **fix 대상에서 제외하고 사람에게 보고**.

### 4-3. 수정 액션 — 기존 규칙으로 되돌림(새 설계 아님)
`kind`별로 이미 있는 규칙을 그대로 재적용한다.
- `radius`/`shadow`/spacing 등 → code-rules §4-1 표기 규약대로 클래스 교체
- 원형화·방향·gap 이탈 → §4-2(명시값 우선)대로 복원
- `color` → 등록 토큰/원문 hex로 교체(§4)
즉 "AI가 어긴 규칙을 규칙대로 다시" 하는 것이며, code가 새로 판단하지 않는다.

### 4-4. 안전장치 & 루프 제어
- **범위 제한**: 짚어진 요소만 패치. 파일 전체 재생성 금지(code-rules §2). 수정 후 빌드 확인(§9).
- **수렴/중단**: 재검증 후 통과면 종료. 같은 항목이 **개선 없이 반복 실패하면 중단**하고 사람에게 보고(무한 루프·발산 방지). 최대 N회.
- **결정론**: 같은 verify 입력 → 같은 수정(재실행 결정론 유지).

## 5. 의존성 (혼자 못 감)

1. **매칭 계약** — 4-2의 위치 특정이 루프의 생명. `data-dk`(앵커) 방식으로 확정할지, verify의 geometry 매칭에 기댈지 **팀 합의** 필요. 앵커면 값 형식(현재 code-rules §4-3은 인덱스 경로, verify #18 match.mjs는 이름 경로 — 불일치)도 통일해야 함.
2. **verify 출력** — 자동 수정은 **속성 단위 기대/실제**(현재 advisory `checks.items`)를 소비한다. 픽셀 게이트만으로는 "무엇을 고칠지"를 알 수 없다. 따라서 style check 매칭이 신뢰도를 확보해야 fix도 신뢰도를 갖는다.

## 6. verify(#18)에 요청할 것

- `checks.items`의 `nodePath`가 **소스 특정 가능한 키**여야 한다(= 매칭 계약 통일과 동일 사안).
- 자동 수정 대상 판별을 위해 각 항목에 **신뢰도/매칭 근거**(strategy, confidence)를 유지해 달라 — 낮은 신뢰도 항목은 fix에서 제외한다.
- 반응형: breakpoint별로 리포트가 분리 저장돼야 한다(현재 `.verify.md`가 breakpoint 없이 저장돼 덮어써지는 이슈 별도 보고).

## 7. 단계 계획

1. **(지금)** 이 문서로 방향 고정 + verify 담당(렉스)에게 6장 요구사항 전달.
2. **매칭 계약 확정** — data-dk 사용 여부/형식 합의. code-rules §4-3을 거기 맞춤.
3. **fix 모드 규칙화** — code-rules에 "verify 리포트 소비 → 위치 특정 → 규칙대로 패치 → 재빌드/재검증" 절을 신설.
4. **수렴 루프 오케스트레이션** — `ddalkak` 스킬에서 code↔verify 반복(중단 조건 포함) 연결.
5. **실증** — sandbox 브릿지로 "일부러 틀린 산출물 → 루프 → 수렴" 재현.

## 8. 미해결 / 결정 필요

- data-dk vs geometry: 위치 특정의 1순위 근거를 무엇으로? (매칭 계약)
- fix 대상 범위: advisory style check 전부인지, 특정 kind(색·radius 등)만인지.
- 자동 수정 vs 사람 확인: 어떤 severity/신뢰도 이상만 자동 적용하고 나머지는 보고할지.
