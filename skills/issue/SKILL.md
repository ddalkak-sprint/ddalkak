# 이슈 등록 — 자동사냥 1단계

대화에서 나온 버그·작업거리를 **컨벤션에 맞는 GitHub 이슈**로 만든다.
컨벤션 SSOT: 프로젝트 루트 `AGENT.md` (제목 프리픽스, 라벨, R&R 표) — 먼저 읽는다.

## 절차

1. **내용 정리** — 사용자가 준 설명·직전 대화 맥락에서 다음을 뽑는다:
   - 버그면: 재현 절차 / 기대 동작 / 실제 동작 / 관련 파일 경로
   - 작업이면: 배경 / 할 일 체크리스트 / 완료 기준
   - 정보가 부족하면 이슈 품질이 떨어지므로 아는 만큼만 정확히 적고, 추측은 "추정"이라고 표기한다.
2. **제목** — `[bug]` / `[work]` / `[doc]` / `feat(<영역>):` 프리픽스 + 한 줄 요약.
3. **라벨** — 관련 파일 경로를 `.github/labeler.yml`의 영역 매핑에 대조해 영역 라벨 결정, 성격 라벨(`bug`/`enhancement`) 추가.
4. **담당 배정** — `AGENT.md`의 R&R 표에서 소유 경로 기준으로 담당을 정한다.
   - 담당의 GitHub 핸들을 모르면 `--assignee` 생략하고 본문에 `담당: <닉네임>`을 적는다.
5. **등록**:
   ```bash
   gh issue create --title "<제목>" --label "<라벨,라벨>" [--assignee <핸들>] --body "<본문>"
   ```
6. **보고** — 이슈 URL을 사용자에게 보여준다.

## 규칙

- **등록만 한다.** 남의 영역 버그면 절대 코드를 고치지 않는다 — 이슈 등록 후 하던 일로 복귀 (AGENT.md 원칙 2).
- 등록 전 `gh issue list --search "<키워드>"`로 중복 이슈가 없는지 확인한다. 있으면 새로 만들지 말고 그 이슈에 코멘트를 남긴다.
- 내 영역 이슈이고 사용자가 바로 처리하길 원하면 `/ddalkak:hunt <번호>`를 안내한다.
