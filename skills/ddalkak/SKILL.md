# 딸깍 (오케스트레이터)

Figma → 코드 전체 파이프라인을 조율한다. 각 단계는 개별 스킬로 위임한다.
기본(`gated`) 모드에서는 **단계 사이마다 결과를 보여주고 [skip / 계속 / 중단]을 확인**하고,
`auto` 모드에서는 게이트를 건너뛰고 끝까지 이어서 돌린다.

## 입력
- Figma URL (필수)
- 추출 모드: `section` | `page` (기본: `page`)
- 실행 모드: `gated` | `auto` (기본: `gated`)
  - `gated` — 단계 사이마다 사용자에게 확인을 받는다(아래 게이트 규칙).
  - `auto` — 게이트를 건너뛰고 끝까지 이어서 돌린다. `딸깍 --auto` 또는 "자동으로 돌려줘"로 지정한다.

## 상태 파일
`.ddalkak/ddalkak.config.json` 에 figma url / mode / runMode / 각 단계 완료·skip 상태를 기록한다.
재실행 시 이 파일을 읽어 이어서 진행한다. `runMode`가 없으면 `gated`로 간주한다.

## 단계 (gated 모드에서는 각 단계 후 사용자 확인)

| # | 스킬 | 입력 → 출력 | 담당 |
|---|------|-----------|------|
| 0 | `ddalkak:design-md` | (팀 컨벤션) → `design.md` | 오픈소스 |
| 1 | `ddalkak:bridge`     | Figma URL → `.ddalkak/bridge/<name>.bridge.json` | 닉·초록 |
| 2 | `ddalkak:plan`       | bridge.json → `.ddalkak/plan/<name>.plan.md` | 퓨리 |
| 3 | `ddalkak:code`       | plan.md (+design.md) → 실제 코드 | 퓨리 |
| 4 | `ddalkak:verify`     | code ↔ figma → `.ddalkak/reports/<name>.verify.md` | 글랜·렉스 |
| 5 | `ddalkak:finalize`   | 검증 리포트 → 마무리 정리 | 글랜·렉스 |

## 게이트 규칙 (gated 모드)
- 각 단계 완료 후: 산출물 요약을 보여주고 다음 중 하나를 묻는다.
  - **계속** — 다음 단계 진행
  - **skip** — 다음 단계 건너뛰기 (이유 기록)
  - **중단** — 현재 상태 저장 후 종료
- 특히 [1] bridge 산출 후에는 사용자의 **검토·확인**을 반드시 받고 [2]로 넘어간다.
- [2] plan 산출 후에는 `validate-plan.mjs` 결과(통과/경고)를 요약에 포함해 사용자가 게이트에서 검토하게 한다.

## 자동 모드 규칙 (auto 모드)
- 단계 사이의 [계속/skip/중단] 확인을 **묻지 않고** 다음 단계로 바로 이어간다.
- 각 단계 산출물 요약은 여전히 출력하되(진행 로그), 응답을 기다리지 않는다.
- 검증기는 그대로 돌린다. 결과에 따라:
  - **경고(warning)** — 요약에만 남기고 계속 진행한다.
  - **차단 실패(error)** — 뒤 단계를 안전하게 이어갈 수 없으므로 그 지점에서 **멈추고** 실패 내용을 보고한다. (예: bridge 추출 실패, `validate-plan.mjs`의 blocking 실패, `validate-code` 게이트 실패)
- [4] verify가 실패/조건부로 나오면 자동 수정 루프를 **자동으로 시작**한다(아래 수렴 루프의 상한·중단 조건을 그대로 따른다). 사용자에게 제안·확인하지 않는다.
- 상한 도달·연속 실패 등으로 루프가 끝나도 통과하지 못하면 남은 실패 항목을 보고하고 종료한다.
- auto 모드에서도 `ddalkak.config.json`에 각 단계 상태를 동일하게 기록해 재개·감사를 보장한다.

## verify → code 수렴 루프
[4] verify가 실패(또는 조건부)로 나오면 [5] finalize로 바로 넘어가지 않고, 사용자에게 **자동 수정 루프**를 제안한다. (auto 모드에서는 제안 없이 곧바로 시작한다.)
- **계속(fix)** 선택 시: [3] code를 **fix 모드**로 호출한다(code-rules §10) — verify 리포트가 소스 위치로 짚은
  불일치만 고친다. 이어서 [4] verify를 재실행한다.
- 통과할 때까지, 또는 **같은 항목이 개선 없이 2회 연속 실패**하거나 상한(기본 3회)에 도달할 때까지 반복한다.
- 중단 시 남은 실패 항목·자동 수정에서 제외된 항목을 사용자에게 보고하고 수동 처리 여부를 확인한다.
- 각 반복의 결과(수정 항목 수, 통과 여부)를 `ddalkak.config.json`에 기록한다.
